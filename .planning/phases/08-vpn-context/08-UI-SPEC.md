# Phase 8 — UI Design Contract: VPN Context

> Visual and interaction contract for Phase 8. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> This phase adds VPN tunnel state to the same three surfaces Phase 7 touched —
> EventDetailView, EventLogExport (JSON), and the privacy toggle scope — plus a
> single computed-label substitution on DashboardView's interface card. No new
> screens. No navigation changes. The contract governs placement, the 6→3 state
> projection, the displayName mapping for all six NEVPNStatus cases, the dashboard
> "Other" → "VPN" substitution rule, nil/empty handling, and privacy redaction.

## State Display Projection (6 → 3)

The model stores the **full 6-state NEVPNStatus** (D-01). The UI projects to the
spec's 3-state vocabulary (VPN-01: connected / disconnected / connecting). The
`displayName` mapping below is **the single source of truth** for every UI surface
that renders VPN state. Plan-phase MUST codify these strings exactly.

| NEVPNStatus | Raw Value | encodingString (JSON) | displayName (UI) | Section visible? | Counts as "VPN active" for dashboard label? |
|-------------|-----------|----------------------|------------------|------------------|---------------------------------------------|
| `.invalid` | 0 | `invalid` | `Invalid` | No | No |
| `.disconnected` | 1 | `disconnected` | `Disconnected` | No | No |
| `.connecting` | 2 | `connecting` | `Connecting` | Yes | No |
| `.connected` | 3 | `connected` | `Connected` | Yes | **Yes** |
| `.reasserting` | 4 | `reasserting` | `Reconnecting` | Yes | **Yes** |
| `.disconnecting` | 5 | `disconnecting` | `Disconnecting` | Yes | No |

**Why `Reconnecting` for `.reasserting`:** the spec vocabulary is
`connected / disconnected / connecting`. `.reasserting` is functionally a
re-establishment of an already-connected tunnel (handover, network change). The
human-readable mapping users will understand is "Reconnecting" — it conveys both
"there is a tunnel" and "it is currently being re-negotiated." This is the only
case where displayName diverges from a literal title-case of the enum name.

**Why `Connecting` (not `Establishing`) for `.connecting`:** matches the
requirement wording in VPN-01 verbatim. Avoid synonyms.

**Why `Disconnecting` is shown but does not flip the dashboard label:** the
section is visible because the tunnel is still measurably present in the event
record (diagnostic value). But for the dashboard's *current* interface label, a
tunnel that is tearing down should not be advertised as "VPN" — the user is
about to lose it. Show "Other" until it's gone, then continue showing "Other"
once disconnected.

**Why `Invalid` and `Disconnected` are hidden in the detail section:** the
section threshold from D-05 is `vpnState ∉ {disconnected, invalid}`. Hiding
"no VPN configured" / "VPN not in use" matches the same principle Phase 7
established — absence of context is communicated by absence of the section,
not by an explicit "Disconnected" row.

## Component Inventory: What Phase 8 Touches

### 1. EventDetailView — New "VPN" Section

**Where:** After the existing `Section("Wi-Fi")` block, before `Section("Probe")`.

The full section order becomes: Event → Network → Cellular → Wi-Fi → **VPN** →
Probe → Location → Duration. VPN sits adjacent to Wi-Fi because both are
network-context layers above the cellular link.

**When to show:** Only when `event.vpnState ∉ {disconnected, invalid}`. Section
hides entirely otherwise — same rule as the Wi-Fi section pattern (D-05). The
visible set is `{connecting, connected, reasserting, disconnecting}`.

**Structure:**
```swift
if let state = event.vpnState,
   state != .disconnected && state != .invalid {
    Section("VPN") {
        LabeledContent("State", value: state.displayName)
    }
}
```

**Layout spec:**
- Section title: `"VPN"` — all-caps acronym, no punctuation. Matches casing of
  existing sections (`"Wi-Fi"`, `"Cellular"`, `"Probe"`).
- Field label: `"State"` — single capitalized noun, no colon, no punctuation.
  Matches the Wi-Fi section's label style (`"SSID"`).
- Value: `state.displayName` from the table above (`"Connected"` / `"Connecting"`
  / `"Reconnecting"` / `"Disconnecting"`).
- Nil handling: when `event.vpnState == nil` (legacy events from before Phase 8,
  or events where capture failed), the section is hidden — same treatment as
  `wifiSSID == nil`. Never show `"—"`, `"N/A"`, or `"Unknown"` in this section.

**Why a new section instead of folding into "Network":** the Network section
already shows path-level facts (`Path Status`, `Interface`, `Expensive`,
`Constrained`). VPN is an overlay on top of that path, not a property of the
path itself. Mixing them blurs the meaning of "Interface". Keeping VPN in its
own section also matches the Wi-Fi precedent and makes the JSON export's
`vpnState` field's semantic level obvious.

### 2. DashboardView — Interface Label Substitution (VPN-02)

**Where:** `connectivityStateCard`, the `.headline` `Text` at DashboardView.swift:180:
```swift
Text(monitor.currentInterfaceType.displayName)
    .font(.headline)
```

**New behavior:** the displayed string is sourced from a new computed helper
(`monitor.effectiveInterfaceLabel` or a free function on the monitor) instead of
`monitor.currentInterfaceType.displayName` directly. The helper returns:

| Condition (in priority order) | Returned label |
|-------------------------------|----------------|
| `currentVPNState ∈ {connected, reasserting}` | `"VPN"` |
| otherwise | `currentInterfaceType.displayName` (unchanged from today) |

**Exact label string:** `VPN` — all caps, no qualifier, no parenthetical state.
The dashboard card shows the *current* interface, not a play-by-play. Detail-view
state lives in EventDetailView's VPN section, not on the dashboard.

**Casing/weight:** `.headline`, `.primary` foreground — visually identical to
"Cellular" / "Wi-Fi" / "Other" today. **No bold override, no italic, no color.**

**Animation/transition for `reasserting`:**
- **None.** No italic, no animated dot, no pulsing, no spinner.
- The dashboard label flips from "VPN" (during `.connected`) to "VPN" (during
  `.reasserting`) — i.e. the user sees no change. This is the right behavior:
  reasserting is a transient sub-state of "the tunnel is up." Drawing UI
  attention to it on the dashboard would create noise.
- Reasserting *is* discoverable via the event detail view (where `displayName`
  reads "Reconnecting") and via the JSON export. The dashboard intentionally
  smooths it.

**Why a computed override (not a new `.vpn` enum case):** D-04 locks this in
to avoid a SwiftData enum migration and to keep `interfaceType` a faithful
record of what `NWPath` reported. The dashboard view is the only consumer that
gets the override; raw `interfaceType` stays addressable for analytics.

**Where the override applies vs. does not:**

| Surface | Uses `effectiveInterfaceLabel`? | Reason |
|---------|--------------------------------|--------|
| DashboardView interface card (line 180) | **Yes** | The whole point of VPN-02 |
| EventDetailView Network section "Interface" row | **No** | Detail view shows ground truth (`interfaceType.displayName`); the new VPN section conveys VPN state separately. Mixing both into one "Interface" cell would erase the distinction. |
| EventListView row (compact list) | **No** | Lists already show event type + timestamp; the row does not show interface today and Phase 8 does not add it |
| JSON / CSV export | **No** | Export carries raw `interfaceType` *and* `vpnState` as separate fields — consumers (Apple's Feedback Assistant reviewers) compute their own derived label |

This "dashboard-only override" rule is intentional and load-bearing. Plan-phase
must not extend the override to other surfaces.

### 3. EventListView — No Change

VPN state does not appear in the event list rows. Same reasoning as Phase 7's
SSID decision: the list row is for scanning event type + timestamp; VPN context
belongs in detail view + export. No row redesign.

### 4. Export — JSON

**New CodingKey:** `vpnState` (camelCase, matching all other export keys).

**Encoding behavior:**
- When privacy mode off (`omitLocation == false`): include `vpnState` as the
  `encodingString` (e.g. `"connected"`, `"reasserting"`) when non-nil and
  `state ∉ {disconnected, invalid}`. **Encode `disconnected` and `invalid` as
  key-omitted** — they carry no diagnostic signal and would inflate JSON
  size for the common "no VPN" case.
- When privacy mode on (`omitLocation == true`): omit `vpnState` key entirely,
  alongside `wifiSSID`, `latitude`, `longitude`, `locationAccuracy` (D-08).

**Why omit `disconnected` and `invalid` from the export instead of writing
them:** parity with the detail-view section visibility rule. The export reflects
the same "non-trivial activity" threshold. A reviewer scanning JSON for VPN
context wants `vpnState` to be present *because* a tunnel was up or transitioning
— not present-but-disconnected on every row. Compactness and consistency.

**Encoding strings (lowercase camelCase, stable identifiers):**

| State | encodingString |
|-------|----------------|
| `.invalid` | `invalid` |
| `.disconnected` | `disconnected` |
| `.connecting` | `connecting` |
| `.connected` | `connected` |
| `.reasserting` | `reasserting` |
| `.disconnecting` | `disconnecting` |

These are the literal NEVPNStatus enum names, lowercased — **deliberately not
projected to 3 states in the export**. The export keeps full fidelity (D-01);
only the UI projects.

**Decode:** String-first / Int-fallback, matching `EventType` / `PathStatus` /
`InterfaceType` precedent. Unknown strings or out-of-range ints decode to nil
(treated as "no VPN context recorded for this event").

**Export metadata envelope:** No changes needed to `ExportMetadata`. The
`locationDataIncluded` flag continues to signal sensitive-fields inclusion;
VPN piggybacks on the same gate (D-08).

### 5. Export — CSV (only if CSV is currently emitted)

> **Note for plan-phase:** Phase 7's UI-SPEC referenced a CSV column update,
> but the Phase 7 SUMMARY only lists JSON export changes. If CSV export is
> live, mirror this column. If CSV is not currently shipped, skip this section.

**New column header:** `vpnState`
**Position:** After `wifiSSID`, before `dropDurationSeconds`.
**Value when nil or `state ∈ {disconnected, invalid}`:** empty string.
**Value when privacy mode on:** empty string (stripped, same as `wifiSSID`).

### 6. Privacy Toggle — Label Update

**Current label:** `"Omit location and Wi-Fi data"` (DashboardView.swift:97)

**Updated label:** `"Omit location, Wi-Fi, and VPN data"`

**Why update the copy explicitly:** the toggle's job is to communicate what gets
stripped. Phase 7 set the precedent of naming each gated category explicitly
("location and Wi-Fi") rather than abstracting to "personal context". Phase 8
follows the same precedent because:
1. The user has previously flagged (memory: project_ui_feedback.md) that the
   privacy toggle scope must be visible in its label, not hidden behind
   abstractions.
2. Adding "VPN" is a 4-character edit that future-proofs against the user
   forgetting that VPN is gated, which would be a regression of the v1.1
   privacy guarantee.
3. The Oxford comma form ("location, Wi-Fi, and VPN") reads better than a
   compound ("location and Wi-Fi and VPN") and is the only form that scales
   if a fourth category is added later.

**Toggle behavior:** No logic change — the existing `omitLocation` AppStorage
key continues to drive location, SSID, and now VPN redaction (D-08). The label
change communicates the expanded scope; no new toggle, no new flag.

## Nil / Empty State Contract

| Surface | Condition | Display |
|---------|-----------|---------|
| EventDetailView | `vpnState == nil` (legacy event, capture failed) | Section "VPN" hidden entirely |
| EventDetailView | `vpnState ∈ {disconnected, invalid}` | Section "VPN" hidden entirely |
| EventDetailView | `vpnState ∈ {connecting, connected, reasserting, disconnecting}` | Section visible, one row: `LabeledContent("State", value: displayName)` |
| DashboardView interface label | `currentVPNState ∈ {connected, reasserting}` | `"VPN"` |
| DashboardView interface label | any other VPN state, including nil | `currentInterfaceType.displayName` (unchanged from today) |
| JSON export, privacy off | `vpnState == nil` or `vpnState ∈ {disconnected, invalid}` | Key omitted (encodeIfPresent + state filter) |
| JSON export, privacy off | `vpnState ∈ {connecting, connected, reasserting, disconnecting}` | Key present with `encodingString` value |
| JSON export, privacy on | any value | Key omitted |
| CSV export, privacy off | `vpnState == nil` or `∈ {disconnected, invalid}` | Empty cell |
| CSV export, privacy on | any value | Empty cell |

> **Design principle (carried over from Phase 7):** Never show a placeholder
> like "Not connected" or "N/A" in the detail view — just hide the section. The
> absence of the section is self-explanatory. This applies symmetrically to
> the JSON export's "no signal" cases.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Section header | `VPN` |
| Field label | `State` |
| Field value — `.connecting` | `Connecting` |
| Field value — `.connected` | `Connected` |
| Field value — `.reasserting` | `Reconnecting` |
| Field value — `.disconnecting` | `Disconnecting` |
| Field value — `.disconnected` | `Disconnected` (defined for completeness; never displayed because section is hidden) |
| Field value — `.invalid` | `Invalid` (defined for completeness; never displayed because section is hidden) |
| Dashboard interface label (when VPN active) | `VPN` |
| Privacy toggle label (updated) | `Omit location, Wi-Fi, and VPN data` |
| Empty state (no events) | Unchanged — handled by existing ContentUnavailableView in EventListView |
| Error state | No new error state introduced — VPN capture is best-effort; nil on failure is silent (matches SSID precedent) |
| Destructive confirmation | None — no destructive actions in this phase |

**Copywriting rationale:**
- **`VPN`** (all caps): standard Apple/industry capitalization. Never "Vpn",
  never "vpn".
- **`State`** (capitalized noun): matches the pattern of `SSID` (capitalized
  acronym) and `Type` / `Time` / `Carrier` elsewhere in EventDetailView. Avoid
  `Status` because that word is already used by the Network section's "Path
  Status" row — using `State` here keeps the two unambiguous.
- **`Connected` / `Connecting` / `Disconnecting`**: title-case the literal
  enum name. Matches user expectation; matches every consumer-facing VPN UI
  on iOS.
- **`Reconnecting`** (not `Reasserting`): the word "reasserting" is internal
  Apple-API jargon (NEVPNStatus.reasserting) and would not parse for a
  reviewer. "Reconnecting" is the correct human-language equivalent.
- **No verb forms with "ing the tunnel"** (e.g. avoid "Establishing tunnel",
  "Tearing down"). Single-word state values match the brevity of every other
  field in EventDetailView.
- **`Omit location, Wi-Fi, and VPN data`**: Oxford comma; "and VPN data" not
  "and the VPN" because the other items are described as data ("location data",
  "Wi-Fi data") and VPN context is data of the same shape.

## Interaction Contract

| Interaction | Behavior |
|-------------|----------|
| Tap event row in EventListView | Navigate to EventDetailView (unchanged) |
| View EventDetailView with VPN-active event (`.connecting/.connected/.reasserting/.disconnecting`) | "VPN" section visible; "State" row shows displayName |
| View EventDetailView with non-VPN event (`nil/.disconnected/.invalid`) | "VPN" section absent — no empty section, no placeholder |
| View DashboardView while VPN tunnel is up (`.connected` or `.reasserting`) | Interface card label reads `VPN` instead of `Other` (or whatever raw `interfaceType` says) |
| View DashboardView during VPN `.connecting` | Interface card label still shows raw `interfaceType.displayName` (e.g. `Cellular` / `Other`) — VPN-02 only flips on connected/reasserting |
| View DashboardView during VPN `.disconnecting` | Interface card label shows raw `interfaceType.displayName` — tunnel is going away, do not advertise "VPN" |
| Toggle privacy switch | Label reads `Omit location, Wi-Fi, and VPN data`; location, SSID, and VPN state all stripped from JSON (and CSV if shipped) |
| Export JSON with privacy off | `vpnState` key present for events where state ∈ {connecting, connected, reasserting, disconnecting}; absent otherwise |
| Export JSON with privacy on | `vpnState` key absent for all events |

## Open Decisions Locked by This Spec

The Phase 8 prompt called out five open UX micro-decisions for the researcher to
lock. Resolutions:

1. **Exact `displayName` strings for all 6 NEVPNStatus cases** → locked in the
   State Display Projection table above. Plan-phase must use these literally.
2. **Exact dashboard label string** → `VPN` (all caps, no qualifier). Casing/
   weight match existing `.headline` `.primary` rendering.
3. **Privacy toggle copy** → updated to `Omit location, Wi-Fi, and VPN data`.
   Explicit category naming, Oxford comma, matches Phase 7 precedent.
4. **Section visibility threshold** → `vpnState ∉ {disconnected, invalid}`
   confirmed (D-05). The visible set is `{connecting, connected, reasserting,
   disconnecting}`. `nil` also hides the section.
5. **Reasserting visual indicator on dashboard** → **none.** No italic, no
   animated dot, no spinner. Reasserting and connected both render as `VPN`
   on the dashboard. Sub-state visibility lives in EventDetailView and JSON
   export, not on the home screen.
