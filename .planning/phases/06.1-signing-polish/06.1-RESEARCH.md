# Phase 6.1 Research: Signing Polish & Warning Fixes

## Issue 1: embedded.mobileprovision Not Found on Device

### Problem
`Bundle.main.path(forResource: "embedded", ofType: "mobileprovision")` returns nil on iPhone 17 Pro Max with iOS 26 and paid Apple Developer Program signing (Team VTWHBCCP36).

### Root Cause Analysis
`path(forResource:ofType:)` searches the bundle's Resources directory and root. On iOS, `embedded.mobileprovision` lives at the root of the .app bundle. However, with post-June 2021 developer accounts and automatic signing, Apple's PPQ (Provisioning Profile Query) verification may affect profile embedding.

### Fix Options

**Option A: Try direct path construction (recommended first)**
```swift
let profilePath = Bundle.main.bundlePath.appending("/embedded.mobileprovision")
let exists = FileManager.default.fileExists(atPath: profilePath)
```
If `path(forResource:ofType:)` fails but the file exists at the direct path, this resolves it.

**Option B: Fallback to build-date heuristic**
For paid developer accounts, provisioning profiles are valid for ~1 year from creation. If the profile file cannot be read at all:
- Extract the app's build date from the executable's modification time or Info.plist
- Estimate expiry as build date + 365 days
- Display with a "(estimated)" qualifier

**Option C: Combined approach (A + B)**
Try to read the profile. If nil, fall back to build-date estimation. This ensures the user always sees meaningful expiry information.

### Recommendation
Option C â€” try direct path first, fall back to build-date heuristic. Keep existing plist parsing logic intact for when the file IS readable.

## Issue 2: Sendable Conformance Warning

### Problem
`EventLogExport.swift:47` â€” `Stored property 'events' of 'Sendable'-conforming struct 'EventLogExport' has non-Sendable type '[ConnectivityEvent]'`

### Root Cause
`ConnectivityEvent` is a `@Model` class (SwiftData). SwiftData model classes are NOT Sendable because they're tied to a ModelContext/actor. `EventLogExport` implicitly conforms to `Sendable` via `Transferable`, but holds `[ConnectivityEvent]`.

### Fix
Mark `EventLogExport` as `@unchecked Sendable`. The struct is only used for one-shot export â€” the events are read on the main actor, packaged into the struct, and consumed immediately by the transfer representation. No concurrent mutation occurs.

```swift
struct EventLogExport: Transferable, @unchecked Sendable {
```

## Issue 3: Deprecated CTCarrier APIs

### Problem
- `EventLogExport.swift:75` â€” `serviceSubscriberCellularProviders` deprecated iOS 16.0
- `EventLogExport.swift:76` â€” `carrierName` deprecated iOS 16.0
- `ConnectivityMonitor.swift:347` â€” same two warnings

### Root Cause
Apple deprecated `CTCarrier` and related APIs in iOS 16 with NO replacement. The APIs still function (return "--" or nil in some cases) but produce warnings.

### Fix
Suppress with targeted deprecation silencing. There is no alternative API â€” this is a known gap in Apple's framework.

```swift
@available(iOS, deprecated: 16.0, message: "No replacement available")
private func captureCarrierName() -> String? { ... }
```

Or use inline suppression:
```swift
let carrierName: String? = {
    if #available(iOS 16.0, *) {
        // Still functional, just deprecated with no replacement
    }
    return CTTelephonyNetworkInfo()
        .serviceSubscriberCellularProviders?
        .values.first?.carrierName
}()
```

## Issue 4: Missing `await` on UIDevice Properties

### Problem
`EventLogExport.swift:83` â€” `UIDevice.current.systemName` and `.systemVersion` are MainActor-isolated in iOS 26 SDK.

### Root Cause
In Swift 6.2 / iOS 26 SDK, `UIDevice.current` properties became MainActor-isolated. The `transferRepresentation` closure runs on an arbitrary executor, so accessing these properties requires `await` or pre-capture.

### Fix
Capture the values before entering the transfer closure, or use `await MainActor.run { }` inside.

Best approach: capture at struct creation time (avoids async complexity in the transfer representation):
```swift
struct EventLogExport: Transferable, @unchecked Sendable {
    let events: [ConnectivityEvent]
    let omitLocation: Bool
    let osVersion: String  // Pre-captured
    let deviceModel: String  // Pre-captured
```

## Issue 5: AccentColor Not in Asset Catalog

### Problem
`Assets.xcassets: Accent color 'AccentColor' is not present in any asset catalogs.`

### Root Cause
The asset catalog has no `AccentColor.colorset`. Xcode expects this for the app's tint color.

### Fix
Add `AccentColor.colorset` to `Assets.xcassets` with a system blue default.
